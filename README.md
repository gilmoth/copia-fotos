# Copia-fotos (Electron + Vite Media Organizer)

## Prerequisites
- Node.js (>= 18)
- npm (comes with Node)

## Development

```bash
# Start Vite and Electron simultaneously
npm run dev
```

## Production Build

To create a standalone, portable Windows executable (`.exe`):

1. **Build the Vite renderer frontend** (generates static files in `dist/`):
   ```bash
   npm run build
   ```
2. **Package the Electron app** (generates the ASAR and `.exe` in `release/`):
   ```bash
   npm run dist
   ```

The final standalone portable application will be located inside the `release` folder (e.g., `release/Copia-fotos Portable.exe`).

This will start Vite (`http://localhost:5173`) and launch the Electron application.

### Features Included:
1. **Media Organizer UI (Spanish Localized)**:
   - Uses `dialog.showOpenDialog` over IPC to select Source, Destination (Photos), and Destination (Videos) folders natively.
   - **Persistent Config**: Remembers the three last selected folders (Source, Photos, Videos) across app runs. The configuration is gracefully saved to the standard Electron user data folder (`userData/organizer-config.json`).
   - **Real Source Folder Scanning**: Clicking **Scan** triggers a recursive filesystem reading process in the Electron main process via `fs/promises`.
     - The main process filters the files by media extensions (photos vs videos) and tags them internally, then sends `scan-progress` async events back to the renderer (`ipcMain.handle` + `event.sender.send`).
     - The Vite renderer receives the live updates over IPC (`onScanProgress`), keeping the UI responsive while updating the scanned count and current filename without freezing the app.
   - **Real File Processing (Copy/Move) with Date Extraction**: Clicking **Start** calculates requirements based on scanned extensions. If there are photos, Destino (fotos) must be set. If there are videos, Destino (vídeos) must be set.
     - Filenames and metadata are parsed using `exifr` to extract the created date. Priority: 1. `YYYYMMDD_HHMMSS` in filename, 2. EXIF `DateTimeOriginal` or `CreateDate`.
     - Valid dates (1990 to next year) are routed differently based on media type:
       - Photos: `Destino Fotos/YYYY/YYYY-MM-DD/`
       - Videos: `Destino Vídeos/YYYY/` (no daily subfolders)
     - Missing or unreasonable dates are routed to a sibling directory next to their respective destinations (e.g., `Destino Fotos- errores de importacion/`).
     - Existing files at the destination won't be overwritten; they are assigned unique suffixes (e.g., `_001`).
     - Progress is relayed continuously to the renderer via IPC (`onProcessProgress`).
     - A live-updating scrollable list inside the renderer shows real-time `COPY`/`MOVE` actions and errors, maintaining the last 200 entries.
     - Can be canceled cleanly at any time by clicking **Cancel**.
     - Generates a rich CSV import log (e.g., `DEST/import_log_...csv`) detailing what was copied or moved, exact timestamps, destination, and any rename or extraction errors. The log writer features robust resource unlocking guarantees and exclusive opening handles (`wx`) to safeguard against Windows `EBUSY` runtime errors natively.
     - **Auto-Open Destinations**: Upon successful completion, natively launches the destination folder(s) directly in Windows Explorer if the user leaves the "Abrir carpetas" checkbox checked. Also offers manual buttons for launching destinations individually at the end of the summary.
   - Updates a real progress bar, file tracking information, and status log box reflecting actual operations.
     - **High Performance Design**: Date-extraction leverages ultra-fast regex early-exit heuristics prior to falling back on binary `exifr` metadata sniffing. To banish UI freezing during huge file ingestion, it utilizes a background asynchronous queue array running concurrently, aggressively caches output references per file-path mapping, and micro-batches the payload arrays bound back into the Vite layer to ensure `100ms` buttery smooth IPC bandwidth.
